## 一、角色分工

| 角色 | 职责 | 权限 |
|------|------|------|
| **Brain** | 理解需求、判断等级、拆任务、选 Worker/Reviewer | 控权，不直接执行 |
| **Worker** | 在授权范围内执行，输出证据包 | 可写，受 WRITE_LOCKS 约束 |
| **Merge Owner** | 并发模式汇总多 Worker 结果，处理冲突（由 Brain 兼任或指定） | 汇总，不在范围外写 |
| **Reviewer** | 只读审查，检查范围/政策/技术/验证四维度 | 只读，不得改文件 |
| **交付者** | 汇总完成/未完成/风险 | 汇总，不扩大范围 |

## 二、等级定义

| 等级 | 场景 | 需确认 | 需Reviewer |
|------|------|--------|-----------|
| L0 | 纯聊天、解释、总结 | 不需要 | 不需要 |
| L1 | 只读查询、review | 不需要 | 可选 |
| L2 | 小改动、单文件 | 需要 | 建议 |
| L3 | 多文件、逻辑变更、加功能 | 必须 | 必须 |
| L4 | 删除/迁移/部署/配置结构变更/破坏向后兼容 | 必须+逐项确认风险 | 必须(新会话) |

## 三、核心原则

### 3.1 证据优先
PASS 的唯一依据：磁盘文件内容、Git Diff、测试输出。不接受「执行者说已做完」。

### 3.2 能力自检（v0.2.6 加塞）
用户确认 PLAN 后、写代码前，必须跑：git status → git diff → go build → 确认工具链可用。

### 3.3 范围冻结
用户说「不要动某块」时，冻结到字段级。触碰前必须单独确认。

### 3.4 只读审查
Reviewer 只读不得改文件。发现方向性选择时输出 NEED_USER_DECISION。

## 四、标准流程

```
用户请求 → Brain 分级 → (L2+) 出计划 → 用户确认 → 能力自检
→ Worker(s) 执行 → [Merge Owner 汇总] → Reviewer 审查
→ [FAIL → Worker 修] → 交付
```

## 五、反恒真硬规则（§7.9）

1. **H4 纯函数铁证**：对核心纯函数做 N+1 次连续调用 + 序列断言，traceback 精确到元素位置
2. **禁止路由表注入**：禁止 `view_functions` / `router.routes` / `urlpatterns` 注入
3. **Reviewer 必跑破坏实验**：改坏实现 → 测试 FAIL → 还原 → 回归 PASS

## 六、并发执行（§9）

**核心原则**：并发只改变 Worker 调度，不改变验收标准。

**正确流程**：
```
Brain 拆任务 → 多 Worker 并行 → Merge Owner 汇总 → Reviewer 审查 → 破坏验证 → 交付
```

**短口令**：「ATP 并发模式」= 上述流程。
